# Release Rollback And Replacement Policy

This policy defines how `mitm_anixops` handles bad release workflow artifacts,
bad tags, and bad public release assets.

Current status:

```text
release-rollback-policy-status=accepted
release-rollback-policy-file=docs/architecture/release-rollback-policy.md
release-replacement-policy=immutable-public-tags-and-assets
release-workflow-publication-status=tag-publish-enabled-after-gates
release-publication-environment=github-release-publication
```

## Scope

This policy applies to:

- GitHub Actions release workflow artifacts;
- future public GitHub Release assets;
- tags matching `v*`;
- checksum, manifest, release-note, and summary evidence generated by
  `.github/workflows/release.yml`.

It does not allow local artifacts to be uploaded as release assets.

## Before Public Publication

If a release workflow artifact is wrong before public publication:

1. Delete or ignore the workflow artifact.
2. Fix the source, workflow, or documentation in a new commit.
3. Let `build.yml` pass for that commit on `main`.
4. Rerun `.github/workflows/release.yml` for the same commit.
5. Use only artifacts, checksums, manifests, and notes generated by the new run.

Do not move a public tag to make a failed workflow artifact look current.

## After Public Publication

Future public release assets and tags are immutable by default:

- Do not replace files attached to a public GitHub Release.
- Do not move a public `v*` tag to a different commit.
- Do not delete and recreate a public `v*` tag to hide a bad artifact.
- Publish a successor tag for corrected artifacts.
- State the superseded release, replacement tag, reason, compatibility impact,
  and rollback path in the replacement release notes.

The only exception is a confirmed credential, malware, legal, or user-safety
incident. That exception requires a manual-intervention entry before any public
asset or tag is removed.

## Required Replacement Evidence

A replacement release must include:

- same-commit `build.yml` success;
- release workflow run URL;
- artifact names;
- SHA-256 sidecars;
- manifest JSON and manifest checksum;
- release notes with compatibility scope, known gaps, and rollback path;
- changelog entry;
- manual-intervention entry when external action was required.

## Current Publication Boundary

The current release workflow builds and validates release artifacts. For `v*`
tag runs, it publishes those workflow-generated artifacts to GitHub Releases
only after same-commit `main` CI, stable release readiness, checksum, manifest,
release-note, existing release, and `github-release-publication` environment
gates pass.

Manual release workflow validation from `main` remains non-publishing. If a
workflow artifact is wrong before tag publication, discard it and rerun the
workflow from a corrected commit. If a public release asset is wrong after tag
publication, keep the tag and assets immutable and publish a successor tag with
replacement evidence.
